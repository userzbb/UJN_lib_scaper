# 🔐 爆破逻辑与进度管理机制 (Cracking Logic & Progress Management)

本文档详细解释了 `UJN_lib_scaper` 项目中密码生成、任务调度、进度保存与断点续传的核心逻辑。

---

## 1. 密码字典生成 (Dictionary Generation)

本项目的目标是针对济南大学图书馆座位预约系统的弱密码规则进行测试。已知密码规则为：**出生日期 (DD) + 出生顺序码 (SSS) + 校验码 (C)**。

### 生成规则
我们通过 `src/core/generator.py` 中的 `generate_dictionary_file` 函数动态生成字典。

一个标准的 6 位密码 `DDSSSC` 由以下部分组成：

1.  **DD (Day)**: 出生日期，范围 `01` - `31`。
2.  **SSS (Sequence)**: 出生顺序码，范围 `000` - `999` (可通过 `--max-seq` 限制)。
    *   **性别校验**:
        *   **男生 (M)**: `SSS` 为奇数 (如 001, 003, ...)。
        *   **女生 (F)**: `SSS` 为偶数 (如 000, 002, ...)。
3.  **C (Check Digit)**: 校验码，范围 `0` - `9`。

### 示例
假设目标是男生，只跑 08 号：
*   生成: `080010` ... `080019` (序列 001)
*   生成: `080030` ... `080039` (序列 003)
*   ...以此类推。

## 2. 任务调度与断点续传 (Scheduling & Resuming)

为了支持海量任务的中断恢复，我们设计了 **"全量生成 + 增量过滤" (Generate Full, Filter Incremental)** 的策略。

### 2.1 为什么不直接生成剩余部分？
虽然只生成剩余字典看起来更高效，但会引入复杂的参数状态管理（例如：如果用户上次用了 `--max-seq 500`，这次用了 `--max-seq 200`，只生成剩余部分可能会导致逻辑混乱）。
因此，我们选择 **始终生成符合当前参数的全量字典文件**，然后在读取时根据数据库进度进行过滤。

### 2.2 过滤逻辑 (Filtering Logic)
在 `src/main.py` 启动时：

1.  **读取进度**: 从 SQLite 数据库 (`crack.db`) 读取该学号在各个日期下的 `last_tried_password`。
    ```python
    progress = {
        "M_01": "014613",
        "M_02": "021005",
        ...
    }
    ```
2.  **流式过滤**: `load_tasks_from_file` 函数逐行读取字典文件。
    *   检查当前行的 `day_key` (如 `M_01`)。
    *   如果该 Key 在进度表中存在，且当前密码 `<=` `last_tried_password`，则 **直接跳过** (Skipped)。
    *   否则，`yield` 该任务给工作线程。

这就是为什么你会看到 `Dict Size: 77500` 但 `Skipped: 76000` 的日志。

## 3. 进度保存机制 (Progress Saving)

为了保证性能，我们没有在每次尝试后都写数据库，而是采用了 **异步批量写入** 机制。

### 3.1 数据库结构
SQLite 数据库 `crack.db` 包含两张表：

1.  **`found_passwords` (结果表)**
    *   记录破解成功的 `username` 和 `password`。
    *   一旦找到，程序下次启动会直接读取此表并显示成功。

2.  **`crack_progress_detail` (进度表)**
    *   **Primary Key**: `(username, day_prefix)`。
    *   **Value**: `last_tried_password`。
    *   这意味我们为每个用户的每一天（甚至每种性别）单独维护一个进度指针。

### 3.2 高水位线逻辑 (High-Water Mark)
由于是多线程并发，线程 A 可能处理 `001`，线程 B 处理 `002`。
如果线程 B 先完成，而线程 A 卡住了，我们不能简单地把进度更新到 `002`，否则如果程序崩溃，`001` 就永远漏测了。

但在本项目中，为了追求速度，我们采用了 **“乐观更新”** 策略：
*   `db_updater_loop` 线程接收所有工作线程的 `FAIL_PASS` (密码错误) 信号。
*   它会定期（每 2 秒或每 50 条）将当前批次中 **字典序最大** 的密码写入数据库。
*   **风险**: 极端情况下（如线程 A 崩溃且未报错），可能会跳过极少量密码。但在 HTTP 爆破场景下，这种概率极低且可接受。

## 4. 显示逻辑 (Display Logic)

为了让用户直观了解进度，我们在控制台输出中区分了“总字典量”和“实际执行量”。

*   **Total Dict**: 根据当前参数 (`-d`, `-s`) 生成的字典总行数。
*   **Skipped**: 因断点续传而跳过的任务数。
*   **Progress**: `Skipped + Completed`。
*   **Speed**: `Completed / Time`。

这样设计确保了进度条总是反映 **全局完成度** (例如 99%)，而不是本次运行的进度。
